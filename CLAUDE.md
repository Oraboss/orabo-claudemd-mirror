## CV/SOP Tool Architecture

Both tools use an identical pattern.

### HTML structure (standalone pages)
`cv-tool.html` and `sop-tool.html` are clean standalone pages ‚Äî no navbar, no footer, no Supabase UMD script. The wrapper `#cv-tool-modal` / `#sop-tool-modal` contains five screen divs: `#cv-form-screen`, `#cv-review-screen`, `#cv-payment-screen`, `#cv-loading-screen`, `#cv-download-screen` (same with `sop-` prefix). All hidden by CSS in `css/cvtool.css`. The inline non-module `<script>` polls `window.cvTool` / `window.sopTool` (100 ms interval, max 50 attempts) then calls `init()`.

### Screen visibility
`showModal()` adds `.active` to `#cv-tool-modal` ‚Üí CSS rule `#cv-tool-modal.active { display: block; }` shows it. `showScreen(id)` removes `.active` from all five screen divs then adds it to the target ‚Üí CSS rule `#cv-form-screen.active { display: block; }` etc. Never use `style="display:..."` on these elements.

### Form rendering
`ALL_SECTIONS = CV_SECTIONS` (6 sections) / `ALL_SECTIONS = SOP_SECTIONS` (7 sections). The question files already include "Your details" (user_name, user_email) as section 0 ‚Äî do NOT prepend a separate DETAILS_SECTION. `renderSection(index)` re-renders the entire `#cv-form-screen` on every navigation, including progress header, Back button, body questions, and Next button.

**Back button (`.tool-back-btn`):** rendered into the form body above the section title on every section. Section 0: label "‚Üê Back to Home" ‚Üí `window.location.href = '/'`. Section 1+: label "‚Üê Back" ‚Üí `saveCurrentAnswers(index)` then `renderSection(index - 1)`. There is no separate nav-area back button ‚Äî do not add one back.

**Next button (`.tool-next-btn`):** accent blue pill (self-contained styles in `css/cvtool.css`). Label is "Next ‚Üí" on all sections except the last, where it reads "Review your answers ‚Üí" (advances to the review screen, not directly to payment). Do NOT use `btn-primary` for this button ‚Äî bare `btn-primary` lacks the `.btn` base class in this context and renders unstyled.

Supported question types: `text`, `email`, `select`, `textarea`, `multiselect`, `conditionalSub` (sub-textarea shown when select hits trigger value), `conditional` (question omitted unless parent answer includes a keyword), `fileUpload` (.txt ‚Üí read into textarea), `large` (tall textarea).

### Payment return flow (sessionStorage bridge)
**Success path:**
1. User completes review screen ‚Üí "Continue to payment ‚Üí" ‚Üí payment screen ‚Üí "Select & Pay" ‚Üí `sessionStorage.setItem('orabo_cv_answers', JSON.stringify(answers))` + `orabo_cv_output_type` ‚Üí `purchaseItem(pkg.type, btn, { customer_email: answers.user_email })`
2. Stripe redirects to `cv-tool.html?payment_success=1&item=cv_review`
3. `init()` ‚Üí `handlePaymentReturn()` checks two-part condition: `payment_success=1` in URL AND `orabo_cv_answers` in sessionStorage
4. Clears URL via `history.replaceState`, removes sessionStorage keys, shows loading screen
5. POSTs `{ outputType: item, formAnswers: parsedAnswers }` to `/api/cv` ‚Äî **field name is `formAnswers`, not `answers`**
6. On success: `showDownloadScreen(data, userEmail)` renders Word download buttons from `data.files.cv_docx` / `data.files.cl_docx`

**Cancel path:**
1. User abandons Stripe checkout ‚Üí Stripe redirects to `cv-tool.html?payment_cancelled=1&item=cv_review`
2. `init()` checks: if neither `payment_success=1` nor `payment_cancelled=1`, wipes sessionStorage (clean start). Otherwise preserves it.
3. `handlePaymentReturn()` detects `payment_cancelled=1` + `orabo_cv_answers` in sessionStorage ‚Üí clears URL, restores answers, shows review screen
4. User can edit answers or proceed to payment again without re-filling the form

No userId passed for anonymous flow ‚Äî backend skips purchases table check when userId absent. Download is Word only (DOCX) ‚Äî no PDF. `generatePdf()` produces blank output for HTML input.

### API response shapes
- `/api/cv` ‚Üí `{ success, result, files: { cv_docx?, cl_docx? }, firstName }` ‚Äî `cv_docx` for cv_review, `cl_docx` for cover_letter_only, both for cv_cover_letter
- `/api/sop` ‚Üí `{ success, result, files: { sop_docx }, firstName }`

| `closePreferencesModal()` | Removes `preferences-modal-open` from `body`. |
| `handlePreferencesSubmit(e)` | Form submit handler ‚Äî validates required fields, reads JWT via localStorage scan, POSTs to `/api/dashboard/preferences`, calls `loadJourneyState()` on success. |
| `handleRetakeQuiz()` | Sets `sessionStorage.orabo_retaking_quiz='1'` then navigates top frame to `/#readiness`; `quiz-retake-handler.js` on index.html detects the flag and auto-starts the quiz. |
| `wireJourneyCtas()` | Wires static `data-cta` buttons + adds delegated click listener on `#dash-journey-panel` for `data-action="open-prefs-edit"` and `data-action="retake-quiz"` (Phase 5). |

| `refreshUsageStats(plan)` | Phase 6 dispatcher ‚Äî if plan='pro' fetches `/api/dashboard/usage-stats` and renders Pro widgets (`#value-this-month-widget`, `#monthly-recap-widget`); if 'free' renders upgrade CTA only and skips the fetch. Called on page load (from DOMContentLoaded in dashboard-journey.js) and automatically via the `updateDashboardForPlan` wrap for mid-session upgrades. |
| `renderValueThisMonthWidget(data)` | Phase 6 ‚Äî Pro only. Populates `#value-this-month-widget`. Shows 30d hours saved, 30d value, and 3-item tool list. Empty state on null/zero data. |
| `renderMonthlyRecapWidget(data, state)` | Phase 6 ‚Äî Pro only. Populates `#monthly-recap-widget`. Shows lifetime hours saved, total purchases, and one nudge line computed from `_oraboLastJourneyState`. |
| `renderValueUpgradeCTAWidget()` | Phase 6 ‚Äî Free users only. Populates `#value-upgrade-cta-widget`. Shows upgrade pitch + "Upgrade to Pro" button that calls `showProModal()`. |

- **Phase 6 widget visibility pattern:** Pro widgets (`#value-this-month-widget`, `#monthly-recap-widget`) carry `data-plan="pro"` in HTML ‚Äî hidden by CSS rule `[data-plan="pro"]:not(.unlocked) { display: none }` in `dashboard-journey.css`. `refreshUsageStats('pro')` adds `.unlocked` to show them. Free-user CTA widget (`#value-upgrade-cta-widget`) has no `data-plan`; hidden by `refreshUsageStats('pro')` adding `.dj-widget--hidden` class.
- **`updateDashboardForPlan` wrap (Phase 6):** `dashboard-journey.js` wraps the inline-defined `window.updateDashboardForPlan` (not modifies it) to fire `refreshUsageStats(plan)` after each plan change. This means `handleReturnFromStripe()` triggering `updateDashboardForPlan('pro')` automatically surfaces the Pro widgets without any additional wiring. The wrap includes a `console.warn` if the function isn't found at wrap time.
- **`GET /api/dashboard/usage-stats` (Phase 6):** JWT-gated, `DASHBOARD_V2_ENABLED` flag. Queries `one_off_purchases` by `email` (not `user_id`). Returns `{ hours_saved_30d, value_purchased_30d, hours_saved_lifetime, value_usd_lifetime, tools_used, total_purchases }`. No caching ‚Äî lifetime totals computed fresh from already-loaded rows. `utils/tool-value-map.js` is the source of truth for `hours_saved` and `value_usd` per tool; `value_usd` must stay in sync with ITEM_PRICES in `routes/stripe.js` (stripe.js is canonical).

- **Preferences modal pattern:** `#prefs-modal` visibility is controlled by CSS class `body.preferences-modal-open .prefs-modal { display: flex }` in `css/dashboard-preferences.css` ‚Äî never inline styles. `PREF_PATHWAY` and `PREF_FIELD` are Brevo text attributes (create in Brevo dashboard if not present). `DESTINATION` is an existing Brevo attribute reused.
- **Retake quiz sessionStorage note:** `handleRetakeQuiz()` MUST use `window.location.href`, not an iframe click. sessionStorage is per-browsing-context ‚Äî a flag set in the top frame IS readable by index.html loaded via `window.location.href` in the same tab, but is NOT readable by an iframe (iframes have their own sessionStorage).
- **`window._oraboLastJourneyState`:** set by `dashboard-journey.js` after each successful `/api/dashboard/state` fetch. Used by `openPreferencesModal` for prefill when editing existing preferences.
- **Iframe close:** dashboard listens for `'tool-closed'` postMessage ‚Üí calls `closeTool()`. `payment-return-*.js` wires close to `handleToolClose()` ‚Äî posts `'tool-closed'` (iframe) or redirects to `dashboard.html` (standalone).
- **Pro upgrade from iframes:** post `'initiate-pro-upgrade'` to `window.top`; dashboard calls `initiateStripeCheckout()`. `wireUpgradeModal` in `stripe.js` follows same pattern.
- **Pro tool cards:** `class="pro-tool"` + `data-plan="pro"`. Tooltip via `[data-plan="pro"]::after`; hidden when `.unlocked` present.
- **`#upgrade-btn`:** hidden by default; shown by `updateDashboardForPlan('free')`, hidden by `updateDashboardForPlan('pro')`.

- Pure HTML/CSS/JS for the frontend ‚Äî no framework
- Keep global styles in `styles.css`; feature-specific styles in `css/<feature>.css`
- Feature JS for `index.html` goes in `js/<feature>.js` (ES modules with `import/export`)

Adding a role: add to all three exports with data for all 18 countries. Adding a country: add a `SALARY_DATA` key for all 35 roles, a `COUNTRIES` entry in `paycalc.js` (with `fxRate`, `taxRate`, `cityId`), a `COL_MONTHLY_USD` entry in `paycalc.js`, a `COUNTRY_DATA` entry in `all-countries-pay.html`, and add its ID to the correct `COUNTRY_GROUPS` array in `paycalc.js`.

- Follow existing BEM-like CSS class naming conventions
- No TypeScript ‚Äî plain JavaScript only
- Do not add comments unless logic is non-obvious
- Every public-facing HTML page must include a complete Open Graph block (`og:title`, `og:description`, `og:url`, `og:image` with `og:image:secure_url`, `og:image:type`, `og:image:width=1200`, `og:image:height=630`, `og:image:alt`) referencing `https://orabo.app/og-image.jpg`, plus `twitter:card=summary_large_image` and `twitter:image`.
